### What Role Node.js Played
Even though the frontend is React, **Node.js** is still involved because modern frontend tooling depends on it.

Node.js is being used here as:
- the runtime that executes build tools
- the environment that runs package commands like `npm install`
- the engine that starts the local dev server

Important distinction:

- **React** = frontend library for building the UI
- **Node.js** = runtime used to manage packages and run the frontend tooling locally
- **npm** = package manager used to install dependencies

Node.js is not yet acting as the `Sprint-Ops Tracker` backend in this phase.  
Right now, it is mostly supporting the frontend development environment.

---

### What `npm` Is
`npm` stands for **Node Package Manager**.

It is used to:
- install libraries
- track project dependencies
- run project scripts

Examples from this phase:
- `npm install`
- `npm install react-router-dom`
- `npm run dev`

These commands all rely on the `package.json` file.

---

### What `package.json` Is
`package.json` is one of the most important files in a Node-based project.

It is the project manifest.

It tells the system:
- the project name
- the project version
- what packages the project depends on
- what scripts can be run

For example, when you run:

```powershell
npm install
```


npm reads `package.json` and installs the listed dependencies.

When you run:

```powershell
npm run dev
```

npm looks in `package.json` for the `dev` script and executes it.

So in simple terms:

* `package.json` = project instructions for npm

---

### What `package-lock.json` Is

`package-lock.json` records the exact dependency versions that were installed.

Why this matters:

* it makes installs more repeatable
* it helps keep environments consistent
* it reduces “works on my machine” drift

You usually do not edit this file manually.

Think of it as:

* `package.json` = what the project needs
* `package-lock.json` = the exact versions npm resolved and installed

### Why We Used Vite

We created the React frontend using **Vite**.

Vite is a frontend toolchain that helps:

* scaffold a React project quickly
* run a fast local development server
* bundle the frontend later for deployment

We used this command:

```powershell
npm create vite@latest frontend -- --template react
```

What that did:

* created a new project inside a folder named `frontend`
* used Vite as the project setup tool
* selected the React template

So Vite gave us the initial React app structure instead of building everything by hand.

---

### Why We Chose JavaScript Instead of TypeScript

At the Vite setup prompt, the correct choice was:

* **React**
* **JavaScript**

That choice was made because:

* it reduces setup friction
* it avoids adding another learning curve immediately
* it lets the focus stay on React structure, routing, and app flow
* it fits the “finishable MVP” mindset

TypeScript is valuable, but for this project phase it would have added more complexity than benefit.

---

### Why the Node Upgrade Was Needed

At first, the local setup failed because the installed Node version was slightly too old for the Vite version that got pulled down.

You originally had:

* `v22.11.0`

Vite required:

* `22.12+`

So the tooling refused to run correctly.

Upgrading Node to:

* `v22.22.0`

resolved that compatibility issue.

This is a normal part of modern development:
sometimes the framework tooling expects a newer runtime version.

---

### Why We Removed `node_modules` and Reinstalled

After the first failed attempt, the local dependency installation was in a bad state.

That is why we removed:

* `node_modules`
* `package-lock.json` when present

and then reran:

* `npm install`

This reset the project dependencies and forced a clean install.

#### *What `node_modules` Is*

`node_modules` is the folder where npm installs all project packages.

It often becomes very large and is not written by hand.

Think of it as:

* the downloaded dependency library shelf for the project

You usually do not edit files inside it.

### What `npm install` Did

When you ran:

```powershell
npm install
```

npm:

* read `package.json`
* downloaded the required packages
* built the local dependency tree
* created or updated `node_modules`
* created or updated `package-lock.json`

This step is required before the project can run.

### What `react-router-dom` Is

We installed routing support with:

```powershell
npm install react-router-dom
```

This package allows the React app to have multiple pages or views without needing multiple separate HTML files.

It supports things like:

* `/`
* `/backlog`
* `/sprint-board`
* `/sprints`

Without it, the app would just be one single screen unless you manually built your own navigation logic.

### What the Local Dev Server Is

When you ran:

```powershell
npm run dev
```

Vite started a **local development server**.

That means:

* the app runs on your machine
* you open it in a browser using a localhost URL
* changes update quickly while developing

This is how most frontend work is done during development.

You are not deploying to AWS yet.

### What We Changed in `main.jsx`

File:

```text
src/main.jsx
```

This file is the frontend entry point.

Its job is to:

* load React
* target the HTML root element
* render the main `App` component
* wrap the app in `BrowserRouter`

#### *Why `BrowserRouter` Was Added*

`BrowserRouter` enables route-based navigation in the app.

Without it, the route components in `App.jsx` would not work.

So this file answers:

* where does the app start?
* what top-level wrapper does the app need?

---

### What We Changed in `App.jsx`

File:

```text
src/App.jsx
```

This file became the main application layout and route map.

Its job is to:

* render the `Navbar`
* define the page routes
* decide which page appears for each URL path

Example mapping:

* `/` → Dashboard
* `/backlog` → Backlog page
* `/sprint-board` → Sprint Board page
* `/sprints` → Sprint Management page

So `App.jsx` is effectively the app’s page switchboard.

---

### What `Navbar.jsx` Is

File:

```text
src/components/Navbar.jsx
```

This is a reusable UI component.

Its job is to:

* show navigation links
* let the user move between pages
* give the app a basic top-level structure

This was separated into its own file because components should be reusable and modular.

That is part of the React mindset:
break UI into smaller pieces instead of putting everything in one file.

### How React Thinks Compared to Traditional Static Pages

A useful mental model is:

#### *Traditional HTML mindset*

* one HTML file per page
* manual linking between separate pages
* page reloads happen between pages

#### *React SPA mindset*

* one application
* route-based views
* UI updates inside the app
* components are reused
* the browser often does not fully reload between page changes

SprintOpsTracker is being built as a **single-page application** using route-based views.

Boilerplate means:

* starter code
* framework-generated defaults
* structural code needed before real feature logic begins

* render setup in `main.jsx`

* framework setup
